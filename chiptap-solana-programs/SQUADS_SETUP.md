# Squads multisig setup (2-of-2 solo)

Goal: replace single-keypair `upgrade-authority` on all 3 deployed programs
with a Squads multisig vault, so a stolen / compromised hot wallet
**alone** can't redeploy malicious bytecode.  Setup is one-time per
network (devnet, then mainnet).

Threat model addressed:
- Hot keypair leaks → attacker can still propose, but can't execute
  (needs cold's signature).
- Cold keypair leaks (USB stolen) → same.
- Both leak → attacker wins, but this is now a coordinated physical +
  digital attack (≫ effort vs single-key compromise).

## Manual setup steps (UI flow, ~10 min)

1. **Funding** — make sure HOT has at least 0.5 SOL on the target network:
   ```bash
   # devnet:
   solana airdrop 2 --url devnet
   # mainnet: send real SOL to it
   ```

2. **Squads V4 UI** — open https://v4.squads.so in the browser where
   Backpack is signed in.

3. **Connect Wallet** — Backpack → "Imported Wallet 1" (= HOT).

4. **Switch network** in the top-right dropdown → **Devnet** (later: Mainnet).

5. **Create a Squad**:
   - Name: `ChipTap Multisig` (or whatever)
   - **Members**: 2
     - Member 1: `Dkq4Vi…CJ5s` (HOT — your current wallet, auto-filled)
     - Member 2: `DMJJSE96A4NbuLumx2ux1NiMfaZB4M1KBhxNtgmWRsLd` (COLD)
   - **Approval threshold**: **2 of 2**
   - Click Create → Backpack popup → Confirm
   - Squads will deploy a multisig account (costs ~0.01 SOL rent).

6. **Copy the "Vault" address** from the dashboard.  Squads V4 distinguishes:
   - `multisigPda` — the config account (governance)
   - `vaultPda` — the address that holds funds AND can sign as
     upgrade-authority (this is the one we want).
   Looks like a normal pubkey, e.g. `GhPnT…Bx7`.

7. **Transfer program upgrade-authority** (HOT signs):
   ```bash
   bash transfer-upgrade-authority.sh <VAULT_PUBKEY> --url devnet
   ```
   Script iterates over treasury / chip_nft / battle_arena and
   transfers each.  Prints current and new authority.

8. **Verify**:
   ```bash
   solana program show <PROGRAM_ID> --url devnet
   # → Authority: <VAULT_PUBKEY>
   ```

9. **Cold-keypair hygiene**:
   - Copy `/root/.config/solana/multisig/cold-backup.json` to a USB drive
     OR import the contents into a hardware wallet (Ledger via
     `solana-keygen recover`).
   - **Delete** the WSL file: `shred -u /root/.config/solana/multisig/cold-backup.json`.
   - Save the address `DMJJSE…RsLd` somewhere visible (the public key is
     fine to share).

## How upgrades work after setup

Single-sig flow (before Squads):
```bash
# Hot wallet alone:
anchor upgrade target/deploy/battle_arena.so --program-id <PID>
```

Multi-sig flow (after Squads):
```bash
# 1. HOT proposes upgrade through Squads UI (Programs tab → Upgrade)
# 2. Plug in USB, restore cold keypair to a fresh path
# 3. COLD approves: open Squads UI in another browser profile, sign-in
#    with cold (one-time), approve the pending tx
# 4. Anyone executes the transaction (1 signature with Squads is enough
#    once threshold is met).
```

Or via the Squads SDK if you want to script everything (we can add a
`squads-upgrade.js` helper later — not blocking).

## What can the hot wallet still do alone?

After SEC-20 + Squads:

| Action | Hot alone? | Notes |
|---|---|---|
| `set_paused` / `set_fee_bps` / etc. (admin ix) | ✅ Yes | These are `owner: Signer` checks, gated by `ArenaConfig.owner` — separate from upgrade-authority |
| `treasury.withdraw` (pull revenue) | ✅ Yes | Same — gated by `owner: Signer` |
| Deploy NEW program (different program ID) | ✅ Yes | Doesn't affect existing programs |
| **Upgrade existing program bytecode** | ❌ No | Requires 2-of-2 |
| Transfer SOL from hot wallet | ✅ Yes | Normal wallet semantics |

Insight: Squads protects against **bytecode rug**, not against admin
abuse.  If hot wallet leaks, attacker can still pause the game, change
fees, withdraw treasury — but **cannot** redeploy malicious code to
drain user balances.

To also protect admin actions, transfer `ArenaConfig.owner` to the
Squads vault (one ix per program, easy to add post-launch).

## Next: when ready for full immutability

After 3-6 months of mainnet with no incidents, run:
```bash
solana program set-upgrade-authority <PID> --final --url mainnet-beta
```
This **permanently** removes upgrade authority.  No one can ever change
program bytecode again.  Final step toward true trustlessness.
